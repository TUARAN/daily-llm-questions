# 日常工作流

> 我如何在日常工作中使用 LLM

## 早晨启动流程

### 1. 回顾和规划（5分钟）

```
今天的主要任务：
1. [任务1]
2. [任务2]
3. [任务3]

可能遇到的挑战：
- [挑战1]

需要 LLM 帮助的地方：
- [具体需求]
```

**使用 Prompt：**
```
基于以下任务列表，帮我：
1. 识别优先级
2. 预估时间分配
3. 指出可能的风险
4. 建议任务顺序

任务：[粘贴任务]
```

---

## 编码工作流

### 1. 功能设计阶段

- [ ] 明确需求和约束
- [ ] 设计 API 和数据结构
- [ ] 评审设计方案
- [ ] 列出测试用例

**LLM 辅助：**
- 快速生成设计草图
- 识别边界情况
- 提供不同实现方案对比

### 2. 实现阶段

- [ ] 先写测试用例
- [ ] 实现核心逻辑
- [ ] 代码审查（LLM 辅助）
- [ ] 优化和重构

**LLM 辅助：**
- 代码片段生成
- 代码审查
- 性能优化建议

### 3. 调试阶段

- [ ] 分析错误信息
- [ ] 制定调试策略
- [ ] 逐步定位问题
- [ ] 验证修复

**LLM 辅助：**
- 分析错误原因
- 提供调试思路
- 建议解决方案

---

## 写作工作流

### 1. 构思阶段

- 主题确定
- 读者分析
- 大纲设计

**LLM 辅助：** 参考 [prompts/writing.md](../prompts/writing.md)

### 2. 初稿阶段

- 快速写下思路
- 不追求完美
- 保持流畅

**LLM 辅助：**
- 提供灵感和角度
- 生成框架内容
- 补充例子和数据

### 3. 修改阶段

- 结构优化
- 内容充实
- 语言润色

**LLM 辅助：**
- 改写优化
- 错别字检查
- 可读性评估

---

## 学习工作流

### 新概念学习

1. 先自己理解和总结
2. 用 LLM 验证和补充
3. 通过例子加深理解
4. 实践应用

**Prompt 模板：**
```
我对 [概念] 的理解是：[我的理解]

请：
1. 指出理解偏差
2. 补充关键点
3. 用类比解释
4. 提供实践建议
```

## 晚间复盘流程

### 1. 总结（5分钟）

```
今天完成：
- [任务1] ✅
- [任务2] ⏸️

遇到的问题：
- [问题1]

明天重点：
- [重点1]
```

### 2. 记录到仓库

- 更新 `questions/` 当月文件
- 补充有用的 Prompt 到 `prompts/`
- 记录实验和失败

---

## 使用技巧

1. **不要打断流程**：先完成，再优化
2. **批量处理**：相似任务一起用 LLM
3. **保持节奏**：LLM 是辅助，不是依赖
4. **持续改进**：记录什么好用，什么不好用
